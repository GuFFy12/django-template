## Рекомендуемое окружение Docker

Для максимальной производительности и избежания узких мест при вводе/выводе файлов, метод запуска Dev Container зависит от вашей ОС:

### Метод А: `Reopen in Container` (Стандартное монтирование)

_Лучший выбор для ОС, которые работают с файловой системой Docker нативно._

- **Linux**: Работает идеально "из коробки" с нативным Docker.
- **macOS**: Стандартный Docker Desktop справится, но для максимальной скорости **настоятельно рекомендуется перейти на OrbStack**.

### Метод Б: `Clone Repository in Container Volume` (Изолированный Volume)

_Обязательно для обхода проблем с производительностью кросс-файловых систем._

- **Windows (Строго обязательно)**: **Не монтируйте** файловую систему Windows. Используйте исключительно команду VS Code **`Dev Containers: Clone Repository in Container Volume`**. Код будет жить внутри изолированной файловой системы Linux, предотвращая тормоза WSL/Windows.
- **macOS (OrbStack строго обязательно)**: Если вы хотите использовать изолированный volume на Mac, вам необходим OrbStack. Docker Desktop при клонировании в volume вместе с Docker Compose выдаст ошибку `mounts denied`. OrbStack обрабатывает эту маршрутизацию нативно.

## Установка и инициализация

1. **Установите Copier глобально**:

```bash
pipx install copier
```

2. **Сгенерируйте проект**:

```bash
copier copy https://github.com/GuFFy12/python-template.git python-template
```

_(Имя проекта внутри шаблона не использует Jinja, так что переименовать внутренние папки нужно будет вручную)._

3. **Инициализируйте Git**:
   Зайдите в папку, сделайте `git init`, закоммитьте всё (включая сгенерированный `.copier-answers.yml`) и запушьте в ваш пустой удаленный репозиторий. Сохранение конфигурации Copier позволит в будущем легко подтягивать обновления шаблона через `copier update`.

4. **Запуск Dev Container**:

- Шаблон предназначен _только_ для VS Code. Установите расширение **Dev Containers**.
- После пуша кода — **удалите локальную папку** (особенно если вы на Windows).
- В VS Code откройте Command Palette (`Ctrl+Shift+P`), выберите `Dev Containers: Clone Repository in Container Volume` и укажите ваш репозиторий.

## Инструментарий и Core-функционал

### Управление зависимостями: `uv`

Быстрый и универсальный инструмент для управления зависимостями, сборки и публикации пакетов. В Dev Container все зависимости управляются через него.

- `uv lock` — фиксация (лок) зависимостей.
- `uv sync --frozen` — строгая установка зависимостей из лок-файла.
- `uv add <package>` — добавление новой зависимости в проект.
- `uv run <command>` — запуск команд в виртуальном окружении.

### Линтинг и Форматирование

- **`lefthook`**: pre-commit хуки для запуска линтеров и форматеров при коммите.
- **`ruff`**: Ультрабыстрый линтер и форматер для Python.
- **`dprint`** (`dprint.json`): Молниеносный аналог Prettier для всех не-Python файлов (JSON, Markdown, YAML). Для добавления нового языка используйте `uv run dprint init`.

### Типизация: `pyright` + `mypy`

По умолчанию типизация настроена строго.

- **`pyright`** используется для быстрого анализа и автодополнения прямо в IDE.
- **`mypy`** выступает финальным проверяющим (анализатором) на этапе CI и pre-commit.
- Если вам нужны менее строгие правила, измените `typeCheckingMode` на `standard` для pyright, и раскомментируйте соответствующие расслабляющие настройки mypy в конфигурации. (Иногда mypy потребуются пакеты-заглушки/stubs для библиотек).

## Структура проекта

### Исходный код и тесты

- **`src/`**: Ваша папка с кодом (переименуйте её под свой проект).
- Содержит `__main__.py` / CLI точки входа.
- Файл `py.typed` сообщает `mypy`, что после сборки ваш пакет содержит аннотации типов (если типов нет — не врите и удалите файл).

- **`tests/`**: Тесты, зеркально повторяющие структуру `src`. `conftest.py` служит для фикстур pytest. Запускать тесты можно прямо из бокового меню тестирования в VS Code.

### Конфигурация Python (`pyproject.toml`)

Центральный файл конфигурации проекта:

- `[project]`: Основная мета-информация и runtime-зависимости. Версия подтягивается динамически через `hatchling`.
- `[project.scripts]`: Точки входа (entrypoints) для CLI.
- `[dependency-groups]`: Группы зависимостей. Группа `dev` ставится в devcontainer (инструменты разработки, тесты, линтеры).
- `[build-system]`: Настроено на `hatchling` и `hatch-vcs` (кладет версию в `_version.py`). _Примечание: если у вас фреймворк вроде Django, который требует своей структуры, блок build-system можно удалить._

### Системные файлы

- **.gitignore** / **.dockerignore**: Мусор не коммитим и в образы не тащим.
- **.gitattributes**: Принудительно ставит `LF` окончания строк везде. Сюда же стоит добавлять расширения больших бинарных файлов для `Git LFS`.

## Настройки VS Code

Все основные настройки уже "затюнены" для удобной работы.

- **`extensions.json`**: **Не трогать!** Все нужные расширения (Python, Ruff, dprint, проверка русской/английской орфографии) прописаны в `devcontainer.json`.
- **`settings.json`**: Оптимизированные настройки IDE. Можете добавлять свои по мере необходимости.
- **`launch.json`**: Настройки отладки. Запуск по **F5**. Обязательно поставьте галочки на перехват Exceptions в панели брейкпоинтов, чтобы при падении IDE сразу показывала место ошибки и состояние переменных.
- **`tasks.json`**: Удобные шорткаты. Вызов через `Ctrl+Shift+P -> Run Task`. Рекомендуется ознакомиться с добавленными задачами.

## Окружение Devcontainer

Конфигурация лежит в папке `.devcontainer`:

- **`.env`**: Переменные для контейнеров (например, `UV_LINK_MODE=copy`, чтобы кэш нормально работал при раздельных директориях). Сюда же можно класть креды для локальной БД.
- **`compose.yaml`**: Стандартный Docker Compose. **Важно:** Если добавляете новые сервисы (например, Postgres), обязательно укажите `network_mode: service:workspace`, чтобы контейнеры видели друг друга.
- **`devcontainer.json`**: Здесь меняем имя проекта и управляем расширениями VS Code. При любом изменении файлов в этой папке не забудьте сделать `Rebuild Container`.
- **`Dockerfile`**: Окружение разработки. Не используйте образы `slim` или `alpine` в качестве базовых для Dev Container — это ваше рабочее место, тут нужны полноценные утилиты! Бинарники копируйте из других образов или ставьте через `apt-get` (использование devcontainer features не рекомендуется, чтобы не размазывать зависимости).
- **`post-create.sh`**: Скрипт, выполняемый после сборки контейнера. Отличное место для запуска миграций БД или других мануальных действий.

## Сборка Production-образа (Dockerfile)

В корне проекта находится глубоко оптимизированный `Dockerfile`, использующий подход **Multi-stage build** (многоэтапная сборка). Это позволяет создать максимально легковесный, безопасный и быстрый образ для развертывания вашего приложения.

Процесс сборки разделен на два ключевых этапа:

### Этап 1: Сборка окружения (Builder)

Главная задача этого этапа — быстро и эффективно собрать виртуальное окружение, используя все преимущества кэширования.

- **Изоляция инструментов:** Бинарные файлы `uv` копируются напрямую из официального образа Astral, поэтому вам не нужно устанавливать их через `pip` или системные пакеты.
- **Умное кэширование зависимостей:** Установка разделена на шаги. Сначала монтируются только `uv.lock` и `pyproject.toml` (через `bind` mounts) для установки сторонних библиотек. И только потом копируется исходный код и собирается сам проект. Это гарантирует, что изменение вашего кода не приведет к переустановке всех библиотек.
- **Кэширование `uv`:** Используются специальные кэш-монтирования Docker (`--mount=type=cache`) для внутренней директории кэша `uv`, что радикально ускоряет все повторные сборки образа.
- **Оптимизация версионирования:** При сборке самого проекта используется фиктивная версия (через `SETUPTOOLS_SCM_PRETEND_VERSION`), чтобы избежать инвалидации кэша Docker при коммитах, которые не меняют зависимости.

### Этап 2: Финальный (Production) образ

Этот этап формирует итоговый контейнер, который поедет на сервер. В нем нет ничего лишнего.

- **Легковесность:** В чистый базовый образ Python копируется _только_ готовая папка `.venv` из первого этапа. Кэши `uv`, инструменты сборки и промежуточные файлы остаются за бортом, что минимизирует итоговый размер образа.
- **Безопасность (Non-root):** Для запуска приложения создается специальный системный пользователь и группа `appuser`. Контейнер работает без root-прав, что является стандартом безопасности в Production.
- **Настройка окружения:** Переменная `PATH` переопределена так, чтобы система по умолчанию видела бинарники из `.venv`. Также установлены стандартные переменные для отключения буферизации вывода и запрета создания `.pyc` файлов (`PYTHONUNBUFFERED`, `PYTHONDONTWRITEBYTECODE`).
- **Точка входа:** Приложение запускается через `ENTRYPOINT`, который указывает на CLI-скрипт вашего пакета _(не забудьте переименовать `python-template` на имя вашего проекта)_.

_Примечание: Базовые образы Python в обоих этапах строго зафиксированы по SHA-256 хешам, что гарантирует воспроизводимость сборки и защищает от внезапных изменений в теге `slim-bookworm`._

## CI/CD (GitHub Actions) и Renovate

В шаблоне нет классических сканеров типа Trivy или CodeQL — вся надежда на своевременное обновление зависимостей.

- **Renovate**: Заменяет Dependabot (который менее гибок). Настроен на жесткий пин (pin) всех версий и зависимостей по SHA-хешам. По умолчанию отключен в CI, рекомендуется установить **Renovate GitHub App**, чтобы не тратить минуты Actions. Обязательно мониторьте PR от Renovate!

### Рабочие процессы (Workflows)

- **Test**: Запускает `pre-commit` для всех файлов, затем прогоняет тесты через `pytest`. Выводит красивый отчет в Summary GitHub Actions. В коде также появятся инлайн-аннотации об ошибках, если тесты упадут. В самом pre-commit (локально) тестов нет, чтобы не замедлять коммиты, они выполняются только на CI.
- **Build Docker**: По умолчанию отключен для обычных веток. Для релиза создайте тег (`git tag v1.0.0 && git push --tags`). Собирает под `x86_64` и `arm64` (через QEMU). Включены семантические теги и защита от пуша тега `latest` (используется major version pin). После сборки происходит **Аттестация образа** — это подтверждает, что образ собран именно на GitHub и не был скомпрометирован.
- **Build Package**: Сборка Python-пакета (`.whl` и `.tar.gz`) через `uv`. Аналогично докеру проводится аттестация. Если вы разрабатываете монолит/веб-приложение (например, Django) — этот воркфлоу можно смело удалить.
- _Публикация на PyPI_ опциональна (нужны токены). Без PyPI пакет всё равно можно установить напрямую: `uv add "git+[https://github.com/user/repo.git@v1.0.0](https://github.com/user/repo.git@v1.0.0)"`.
